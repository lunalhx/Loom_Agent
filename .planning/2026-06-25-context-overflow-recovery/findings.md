# Findings

- Existing recovery order is larger-model fallback before reactive compact.
- `contextFallbackAttempted` is local to one `ModelCallNode#doApply` invocation and is not checkpointed.
- `reactiveCompact()` reuses `summaryCompact()` and has no target-size result.
- Gateway preflight reports prompt/context overflow as `MODEL_CAPABILITY_MISMATCH`, bypassing overflow recovery.
- Checkpoint snapshots already persist `reactiveCompactAttempts`, current model, and fallback reason.
- Approval pause/resume provides a proven terminal-with-checkpoint pattern for recoverable user interaction.
- `agent_run.status` is a string column, so `WAITING_USER_INPUT` needs no schema column migration.
- Baseline `DefaultAgentLoopServiceTest`: 26 tests pass.
- `ContextWindowManager` can accept model/budget/trace collaborators through a backward-compatible overloaded constructor; existing tests use the three-argument constructor.
- `ResilientModelGateway` preserves an explicitly supplied capability and purpose, so context summary calls can use a dedicated purpose/capability without changing the gateway port.
- Root context already starts with a `USER_TASK` dynamic entry in `DefaultAgentLoopService#toContext`.
- The existing checkpoint repository is sufficient for user-input resume; no additional pending-input store is required.
- `AgentCodeController` already centralizes Bean Validation and SSE lifecycle; the new input endpoint can mirror approval resume without introducing a new transport pattern.
- Terminal gate nodes are checkpointed through existing AFTER_NODE/STOP hooks, so `UserInputGateNode` can persist a resumable state without setting an error code.
- Review found fallback budget admission was still checking the pre-compaction `currentPrompt`; it should use the compacted dynamic context instead.
- Explicit context fallback success can overwrite the stored `context_overflow` reason with a null provider fallback reason; preserve the existing reason when the result has none.
- Deep-summary merging should recursively summarize packed summaries rather than concatenating all summaries into a potentially oversized final request.
- Existing DTO validation tests are lightweight Bean Validation checks; user-input semantics are more meaningfully covered through the service resume tests.
- Final status review found ABORT could still be labeled `waiting_user_input` in trace metrics because the waiting stage was checked before the error code.
- README and the design guide already document Agent SSE endpoints, so the new recovery event and input endpoint should be added there rather than creating a separate API document.
